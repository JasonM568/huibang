## D-006｜薪資「足月」門檻以 30 天為準（`Math.min(30, 該月實際天數)`）
- **日期**：2026-07-10
- **背景**：日薪固定用「足月應領總額 ÷ 30」計算（台灣慣例，一個月當 30 天），但「是否足月」的判斷卻是拿計薪天數比對**該月實際天數**（`payDays >= monthLastDay`，28~31）。Jason 回報 31 天月份填計薪 30 天時被誤判為「不足月」，要填 31 才對。
- **決策**：足月門檻改為 `payDays >= Math.min(30, monthLastDay)`。大月（31 天）門檻降為 30（計薪 30 或 31 天都算足月）；小月（30 天）門檻 30；二月（28/29 天）取實際天數，**做滿整月即算足月領全額**（已與 Jason 對齊）。
- **為什麼**：日薪分母固定是 30，「足月」的語意就該是「做滿一個月 = 30 天」，而非日曆天數。二月不足 30 天但做滿整月仍應領全額，故對短月取實際天數為門檻，避免永遠被當不足月。
- **影響**：改動 5 處同一行邏輯（`salary/new`、`salary/[id]` 編輯+檢視、salary POST、salary PATCH）。commit `c655f68`。既有紀錄不回頭重算，只有被重新編輯存檔（走 PATCH）時才依新邏輯重算。

## D-005｜`228c27a` 全刪除災難：以 reset 到最後健康 commit + force-with-lease 復原
- **日期**：2026-06-22
- **背景**：開工檢查 git 時發現 `git ls-tree -r HEAD` 回傳 **0 檔案**。逐筆比對 commit tree：`7d81ac2`（及更早）= 184 檔健康，但 **HEAD `228c27a`（"公司資訊改 no-store"）tree = 0 檔，且已 push 到 `origin/main`**。即 CLAUDE.md 警告的 zsh 中括號 glob → `git add 'path/[id]/file'` 連鎖 → 整個 repo tree 被刪光並推上 GitHub。工作目錄檔案全在（僅變未追蹤），`228c27a` 想做的 no-store 改動也安然在工作目錄。
- **決策**：
  1. `git read-tree 7d81ac2` 對齊 index 看清真實差異（確認工作目錄相對最後健康 commit 只差 4 檔：no-store 修復 + 今天新功能），不貿然 `git add -A`（避免把刻意不版控的 .md / public 圖 / zip 一起收進去）。
  2. `git reset --soft 7d81ac2` 把 main 退回最後健康 commit、丟棄空 tree 的 `228c27a`，工作目錄不動。
  3. 只 stage 4 個合法變更（CompanySettingsTab + 請款單詳情頁的 no-store、報價單列印頁匯款資訊、新請款單列印頁），重做乾淨 commit `361f771`（tree 185 檔）。
  4. `git push --force-with-lease origin main` 取代 origin 的壞 commit。**Jason 自己用 `! git push…` 跑**（Claude Code auto-mode classifier 會擋 Claude 直接對 main force-push，合理）。
- **為什麼**：`reset --soft` + force-with-lease 是 CLAUDE.md 既定的災難復原法；以「工作目錄為真相」重建，既救回 184 檔又不丟 `228c27a` 的真實意圖（no-store），且把今天功能一起帶上。`--force-with-lease` 確認 origin 仍停在 228c27a 才覆蓋，安全。
- **影響**：origin/main 從 228c27a → 361f771（185 檔健康），Vercel 重新部署。**過程踩到第二個坑**：`.git/refs/remotes/origin/main 2`（macOS 重複檔）讓 `fetch` 報 "bad object" 並卡住 `--force-with-lease`（不帶參數時要讀此 remote-tracking ref）→ `rm` 掉該壞檔、重新 fetch 後才推得動。**教訓**：(1) 每次開工先 `git ls-tree -r HEAD --name-only | wc -l` 核對檔案數；(2) 嚴守 CLAUDE.md 安全 git 流程，中括號路徑務必引號且推前查 `D` entries；(3) `.git/refs` 偶有 macOS ` 2` 重複檔會壞 ref，fetch 報 bad object 時先去 `.git/refs/remotes/origin/` 找帶空白的重複檔刪掉。

## D-004｜分期請款用「分階段」模型，並以報價單 `partial` 狀態追蹤
- **日期**：2026-06-19
- **背景**：第一版分期以「Modal 列數 > 1」判定是否分期、且強制一次湊滿 100%。Jason 實際是**分階段請款**（第一期 40% 先開、之後再開二/三期），結果單列「第一期 40%」被當成全額轉換、忽略期別與百分比，回報「分期根本做假的」。
- **決策**：
  1. **分階段模型**：一張報價單可分多次轉換，每次依百分比開「第N期」請款單；後端依該報價單「**已請款比例**」累計（全額單計 100%、分期單以 `installment_percent` 累加），新一期不得使累計超過 100%，補足 100% 的那期吸收四捨五入尾差，確保各期金額總和精確等於報價單總額。期別序號 `installment_no` 跨次累加，單號加 `-P{n}`。
  2. **不強制一次 100%**：可只開第一期，報價單保留可繼續開下一期。
  3. **報價單新增 `partial`（分期請款中）狀態**：累計 <100%→`partial`、=100%→`invoiced`；刪請款單後依剩餘比例回推 `invoiced`/`partial`/`sent`（0% 解鎖回 sent）。有請款單（partial/invoiced）即鎖住「編輯/改狀態/刪報價單」，未滿 100% 仍可「繼續開立請款單」。
  4. 請款單詳情頁以單行「第N期款項」呈現（GET 偵測 `installment_no` 時回傳合成單行，否則沿用報價品項）。
- **為什麼**：符合工程專案里程碑請款（簽約款/驗收款/上線款）的真實流程；以「已請款比例」為單一真相來源，分多次開立也能保證金額對齊、狀態正確，且守住「已開請款單的報價內容不可亂改」。
- **影響**：`invoices` 加 `installment_no/label/percent`、`invoice_number` 放寬 `varchar(40)`；既有全額請款單（installment 欄位為 null）視為 100%，行為不變。既有資料已回填狀態（BML→partial、一張漏更新的→invoiced）。

## D-001｜工作脈絡用「關鍵字 + CLAUDE.md 指令」觸發，不用 hook
- **日期**：2026-06-09
- **背景**：要做收工存檔、開工接軌的自動化。
- **決策**：把「收工 / 開工」協定寫進 `CLAUDE.md`，靠關鍵字觸發，不另設 settings.json hook。
- **為什麼**：CLAUDE.md 每次對話自動載入，透明、可控、零額外設定，立即可用；hook 較複雜且不易檢視。
- **影響**：觸發依賴關鍵字（收工／今天先這樣／開工／開始做）。若日後想要「真・自動」可再升級為 Stop hook。

## D-003｜報價折扣下放到「項次」層級，`quotes.subtotal` 改存折後小計
- **日期**：2026-06-15
- **背景**：原本折扣是整單一個折扣率、套在總小計上；Jason 要求改成每個報價項次各自一個折扣率，且項次小計直接顯示折後金額（16,800 折 30% → 11,760）。
- **決策**：
  1. `quote_items` 新增 `discount` 欄；項次 `amount` 改存折後金額 `round(單價×數量×(1−折扣%))`。
  2. `quotes.subtotal` 由「折前小計」改存「**折後小計（未稅）**」；稅額、總計皆以折後為基礎。整單 `quotes.discount` 欄位停用但保留（相容舊資料）。
  3. 報價頁的「折前小計／折扣總額」改由項次即時推導顯示（折扣列保留為資訊）。
- **為什麼**：請款單沿用報價品項、並複製 `quotes.subtotal` 顯示「小計＋稅額＝總計」（無折扣列）。把 subtotal 定義為折後，請款單自動保持一致，**完全不需修改請款模組**，也守住「已轉請款單不可更動」的要求。
- **影響**：只新增欄位（預設 0），不做資料遷移、不回頭重算 → 既有資料零影響，未來轉換才走新邏輯。唯一一張舊整單折扣的測試報價單折扣顯示會跑掉（可重編或刪除）。整單折扣輸入框已從新增/編輯頁移除。

## D-002｜報價單轉請款的狀態更新放在後端唯一入口，不靠前端 PATCH
- **日期**：2026-06-09
- **背景**：報價單轉請款後狀態一直停在「草稿」。詳情頁按鈕原本是轉換成功後，再由前端額外 PATCH 報價單 status，且未檢查成敗。
- **決策**：改在 `POST /api/admin/invoices`（所有轉換的唯一入口）建立請款單後，伺服器端直接 `update quotes set status='invoiced'`。前端那段重複 PATCH 保留不動（無害）。
- **為什麼**：所有轉換必經此入口，保證狀態一定更新，不依賴前端行為或網路成敗。
- **影響**：只對之後的轉換生效；舊的「已轉請款但仍草稿」報價單需另跑一次性 SQL 才會回補。狀態碼用 `invoiced`，顯示「轉請款單」。
